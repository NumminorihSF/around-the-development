Почему вы занимаетесь разработкой?

Возможные причины, почему вы в IT

  • Удовольствие от написания кода

За это не платят денег

Удовольствие от кода

  • Бизнесу нужно решение проблем
  • Иногда решать эти проблемы не приятно
  • Работа с чужим кодом может приносить боль
  • Хочешь написать идеальный код - ты его никогда не допишешь

Но! Если не получаешь удовольствия от работы, то будет тяжело

Удовольствие от задач

  • Бизнесу все еще нужно решение его проблемы
  • Проект никогда не состоит только из интересных задач
  • Часть задач - малоинтересные баги

Машины лучше людей

  • Работать с людьми придется
  • Рост заработной платы и карьерный рос завясят от навыков общения
  • Мы работает для людей

Деньги / печеньки

  • Нужно вкалывать
  • Нужно постоянно учиться
  • Велика вероятность перегореть
  • Хотите ли вы потратить жизнь на работу ради этого?

Ради чего пишу код я

  • Ради процесса

  • Для решения бизнес-задачи

Разработка - не только кодирование

Митинги

Stand-up митинги

  • Выяснить текущий прогресс команды
  • Выяснить текущие проблемы, договориться, когда будете решать
  • Планирование, уточнение общих вопросов

Как можно испортить митинг

  • Сдвинуть время митинга
  • Не придти
  • Опоздать
  • Втыкать в телефон/компьютер
  • Витать в своих мыслях
  • Оставить микрофон включенным, находясь в шумном месте

Как можно испортить митинг

  • Пытаться везде вставить свои 5 копеек
  • Уводить обсуждение в сторону от основной темы
  • Пытаться решить все вопросы здесь и сейчас
  • Слишком углубляться в тему
  • Не давать достаточно информации
  • Общаться техническим языком, если на митинге присутствует нетехнарь

Code review

Code review

  • Инструмент обучения новичков
  • Свежий взгляд на код
  • Система документирования кода
  • Процесс контроля:
    • правильности выполнения задачи
    • соблюдения договоренностей
    • поддерживаемости решения

Улучшить процесс CR

  • Договоритесь о форматировании на берегу
  • Машины должны страдать
  • Не задерживайте ревью
  • Не пытайтесь сделать весь код идеальным
  • Используйте примеры с кодом

Улучшить процесс CR

  • Используйте мы вместо ты
  • Будьте вежливы
  • Принципы > мнения
  • Забудьте, кто писал код

Написание документации

Документирование кода

  • Добавляет ясность в код
  • Позволяет использовать код без его чтения
  • Позволяет удобнее писать код

Проблемы документирования

  • Нужно больше времени на написание фичи
  • Для большинства программистов это скучно
  • Сложно писать хорошую документацию
  • Где хранить?
  • Тяжело поддерживать документацию в актуальном состоянии
  • Код все равно придется читать

Почему нужно ее писать

  • Самодокументируемый код будет разный для разных людей
  • Код, который мы пишем - не наша собственность
  • Отсутствие комментариев - самый простой способ указать что код плохой
  • Экономит наши деньги
  • Новый человек быстрее вливается в проект

Unit-тесты

Проблемы unit-тестирования

  • Нужно больше времени на написание фичи
  • Сложно писать хорошие тесты
  • Сложно писать код, который удобно/легко тестировать
  • Нужно тратить время на поддержку тестов
  • Описание теста валидируется только руками
  • Coverage в 100% не гарантирует ничего
  • Никак не защищает от ошибок интеграции модулей

Хороший unit-тест

  • Позволяет проверять модули без интеграции в приложение
  • Уменьшает вероятность регрессий
  • Является примером использования модуля
  • Проверяет конкретный use-case
  • Значительно упрощает рефакторинг
  • Экономит время

Обучение и sharing знаний

Я знаю, что ничего не знаю

Не будем никого обучать

  • Начинающий разработчик всегда совершает больше ошибок, поэтому не стоит его нанимать
  • Повышение затрат времени и внимания старших разработчиков
  • Не каждый старший разработчик хочет/может заниматься обучением
  • Большая часть более-менее детальной информации все-равно будет забыта
  • То что я рассказываю - очевидно всем
  • Те, кого я обучу меня заменят

Зачем обучать и делится знаниями

  • Команда сможет делать больше задач
  • Команда сможет делать более сложные задачи
  • Экономия времени
  • Снижение рисков
  • Обучить правильному подходу с нуля - проще, чем переучивать

Зачем обучать и делится знаниями

  • Упорядочивание собственных знаний
  • Выработка более глубокого понимания темы и выявления пробелов
  • Выработка у обучаемых правильного подхода
  • Возможность получить конструктивную критику
